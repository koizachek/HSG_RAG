# Documentation of the Accuracy Test for the RAG Chatbot

## 1. Objective of the Evaluation

The objective of this evaluation was to systematically assess the factual correctness of the answers generated by the RAG-based chatbot before deployment. The chatbot answers questions about the EMBA HSG, International EMBA HSG, and emba X programmes. Since the answers may be used by prospective students, it is important to verify whether the chatbot provides reliable, up-to-date, and programme-specific information.

A target value of at least 95% factual correctness was defined. In addition to measuring the overall accuracy, the evaluation was also intended to identify recurring error types and areas where the chatbot required improvement.

## 2. Structure of the Test Catalogue

The evaluation was based on a catalogue of 100 test questions. The questions covered central information needs of potential users, including:

- tuition fees and financing options
- application deadlines and admission requirements
- programme duration and study format
- curriculum, modules, and credits
- contact information and contact persons
- programme-specific differences between EMBA HSG, International EMBA HSG, and emba X
- German and English user queries

The test catalogue was designed to include both simple factual questions and programme-specific distinction questions. This made it possible to assess whether the chatbot retrieved the correct information from the available sources and assigned it to the correct programme.

## 3. Technical Execution

The test questions were not entered manually via the user interface. Instead, they were executed automatically using a Python script that directly called the bot’s agent chain. This ensured that all questions could be tested in a reproducible and consistent way.

The script reads a CSV-based question catalogue, sends each question to the `ExecutiveAgentChain`, and stores the chatbot response together with relevant metadata in a results file. The stored information includes, among others:

- the original question
- the chatbot response
- the detected response language
- the session mode
- the session ID
- the model name
- the commit hash
- timestamps of the test run
- cache settings

By default, the test was executed in isolated session mode. This means that each question was processed in a fresh session. This prevents previous questions from influencing later answers.

The cache was disabled by default. This ensured that the answers were not reused from previous test runs, but were generated again through the current chatbot pipeline.

## 4. Evaluation Method

After the automated generation of chatbot responses, the answers were manually evaluated. Each answer was checked for factual consistency with the official programme information. The evaluation used the following categories:

| Rating | Meaning |
|---|---|
| Correct | The answer is factually correct and answers the question sufficiently. |
| Partially correct | The answer is mostly correct, but incomplete, imprecise, or not specific enough. |
| Incorrect | The answer contains incorrect information or does not answer the question sufficiently. |

In addition, a numerical score was assigned to each answer. This score allowed for a more fine-grained assessment than the categorical rating alone. Partial deductions were made, for example, when concrete numbers, specific months, contact information, or payment details were missing.

## 5. First Test Run

In the first test run, 100 chatbot responses were evaluated. The results showed that the chatbot was already able to answer many questions correctly, but still had noticeable weaknesses in several areas involving detailed factual information.

### Results of the First Test Run

| Category | Number |
|---|---:|
| Correct | 65 |
| Partially correct | 14 |
| Incorrect | 20 |
| Not evaluated / empty | 1 |

The strict accuracy, where only fully correct answers were counted as successful, was therefore 65%. When using the numerical scores, the weighted score was approximately 71.38%.

### Observed Weaknesses in the First Test Run

The first test run revealed several recurring issues:

- partially outdated information
- missing or imprecise information on fees and payment rates
- missing concrete numbers
- incorrect or incomplete information on application deadlines
- missing credit information when questions referred to programme credits
- overly general answers that referred the user to the Admissions Team
- answers that were generally in the right direction but not precise enough

These results showed that the first tested version of the chatbot did not yet reach the target reliability level of 95%.

## 6. Improvements After the First Test Run

After the first test run, targeted adjustments were made to the chatbot. The evaluation therefore served not only as a measurement of answer quality, but also as a basis for concrete improvements.

The changes aimed to improve factual accuracy and completeness. Particular attention was given to information areas where errors or uncertainties had been identified during the first test run, such as fees, deadlines, credits, and programme-specific details.

After these adjustments, a second test run was conducted to check whether the answer quality had improved measurably.

## 7. Second Test Run After Adjustments

In the second test run, another set of 100 responses was evaluated. The results show a clear improvement compared to the first test run.

### Results of the Second Test Run

| Category | Number |
|---|---:|
| Correct | 90 |
| Partially correct | 5 |
| Incorrect | 5 |

The strict accuracy, where only fully correct answers were counted, was 90% in the second test run. This means that the chatbot significantly increased the number of fully correct answers.

Based on the numerical scores, the weighted score was approximately 96.1%. However, it should be noted that some answers marked as “Incorrect” still received a score of 1 in the evaluation file. These entries should be cleaned up before the final reporting. If the rating category “Incorrect” is used as the decisive criterion, the stricter success rate is 90%.

## 8. Comparison of the Two Test Runs

| Metric | First test run | Second test run | Change |
|---|---:|---:|---:|
| Number of test questions | 100 | 100 | — |
| Correct | 65 | 90 | +25 |
| Partially correct | 14 | 5 | -9 |
| Incorrect | 20 | 5 | -15 |
| Strict accuracy | 65% | 90% | +25 percentage points |
| Weighted score | approx. 71.38% | approx. 96.1% | +24.72 percentage points |

The comparison shows that the answer quality improved substantially after the adjustments. The number of incorrect answers decreased from 20 to 5, while the number of fully correct answers increased from 65 to 90.

This demonstrates that the changes made after the first evaluation had a measurable positive effect on the factual correctness of the chatbot.

## 9. Error Analysis of the Second Test Run

Although the second test run showed a significant improvement, some weaknesses remained. These mainly concerned the level of detail and completeness of individual answers.

Recurring issues included:

- credit information was sometimes not mentioned, although it was requested
- payment rates or concrete financing details were sometimes not provided directly
- in some cases, the chatbot referred the user to the Admissions Team instead of giving the information directly
- some time-related answers were not precise enough, for example when the exact month was missing
- for contact-related questions, it was sometimes unclear whether concrete contact information should be provided directly or whether a referral was sufficient

These remaining errors are less fundamental system failures and more related to the desired level of answer detail. Before deployment, it should be clearly defined whether the chatbot should always provide concrete factual details directly or whether referral to the Admissions Team is preferred in certain cases.

## 10. Assessment of Target Achievement

The original target was a factual correctness of at least 95%.

Depending on the calculation method, the result can be interpreted differently:

- Under a strict evaluation, where only fully “Correct” answers are counted, the chatbot achieved 90% in the second test run.
- Under a weighted evaluation based on the assigned numerical scores, the chatbot achieved approximately 96.1%.

Because some entries marked as “Incorrect” still received a score of 1, the evaluation file should be cleaned before the final analysis. For a conservative assessment, the strict accuracy of 90% is more meaningful. For a more differentiated assessment that gives partial credit to partly correct answers, the weighted score indicates that the chatbot is close to or above the target value.

## 11. Recommendations for Further Improvement

The following measures are recommended for further optimisation:

1. **Clean up the evaluation file**  
   Entries marked as “Incorrect” but assigned a score of 1 should be reviewed and corrected.

2. **Strengthen answer rules for concrete facts**  
   When users ask for numbers, credits, deadlines, fees, or payment rates, the chatbot should provide these facts directly if they are available in the sources.

3. **Define a clear rule for Admissions Team referrals**  
   It should be specified when referring users to the Admissions Team is sufficient and when concrete information must be included directly in the answer.

4. **Improve source coverage for credits and programme details**  
   Since credit-related information appeared repeatedly as a problem, it should be checked whether these details are clearly available in the source material and easily retrievable by the system.

5. **Continue using the test catalogue as a regression test**  
   The 100-question catalogue should continue to be used as a regression test whenever changes are made to the chatbot, the source documents, or the prompting strategy.

## 12. Conclusion

The evaluation shows that the RAG chatbot improved significantly after the adjustments. In the first test run, the strict accuracy was 65%. After the changes, the strict accuracy increased to 90% in the second test run. At the same time, the number of incorrect answers decreased from 20 to 5.

The weighted score of the second test run was approximately 96.1%, but this value should be interpreted carefully because of a few inconsistent score entries in the evaluation file. Overall, the second test run shows that the chatbot achieved a substantially higher answer quality and became much more reliable.

Before deployment, the remaining issues regarding credits, concrete numbers, payment rates, and detailed factual information should be reviewed. After these final improvements, the test catalogue should be executed again to confirm whether the 95% target is also reached under the strict evaluation method.
